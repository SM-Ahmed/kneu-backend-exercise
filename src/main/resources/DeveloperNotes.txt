Overall Approach:
1. Focus on making a completed flow with emphasis on getting DTO/database structures right so that it can be easily extended in the future.
2. Test Driven Development: after defining the data structures, I write an end to end test which covers the main functionalities I need to implement. Once code is complete, this test passes.
3. Copilot used to reduce manual work (e.g. writing transformers, boilerplate code, tests etc.). I just defined the data structures, data architecture and test scenarios to prompt Copilot to work. Afterwards, I adjusted output appropriately.
4. Copilot usage was sparing (due to being on free plan).

Assumptions:
1. One prescription can have multiple medicines (or this may happen in the future)
2. Prescription administration time is an optional field. Assume that it is daily medication, so I do not need to specify frequency etc (e.g. via cron expression)

Trade-offs (due to time constraints):
1. Many features left unimplemented (see improvements below)
2. Validation and error handling is very basic. Ideally, I would use separate validator classes and aspects

Improvement points:

1. OpenAPI schema: formalise external API schema, generate external pojo classes from it, have separate dtos for external api requests and internal dtos, easier validations
2. Database: implement proper changelog handling (e.g. Liquibase), transactions etc.
3. Logging: implement proper logging (e.g. SLF4J), log levels, log formatting, log rotation etc.
4. Junit: add 100% junit test coverage on all lines of code + branches
5. Modularity: separate dao module handling jpa features so core module doesnt bother with entity objects at all.
